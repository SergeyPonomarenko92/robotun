# Вступ

Robotun — це гібридна платформа, що поєднує соціальну взаємодію у стилі Instagram (профілі, стрічки, сторіс, портфоліо), маркетплейс оголошень у стилі OLX (пропозиції та запити на товари/послуги) і прозору систему відгуків у стилі Rozetka. Мета — дати користувачам єдиний канал для пошуку виконавців, покупки/продажу товарів та прийняття рішень на основі реального соціального доказу (портфоліо + відгуки).

## 1. Призначення та сфера застосування

### 1.1 Призначення

Платформа виступає посередником між:

- **Постачальниками послуг** (майстри з ремонту, сервісні бізнеси, ремісники, локальні магазини — наприклад, пекарні).
- **Продавцями товарів** (фізичні та юридичні особи).
- **Покупцями та замовниками** (люди, які шукають виконавця, товар або послугу).

### 1.2 Сфера застосування

Ця специфікація визначає **архітектурні межі, ключові домени, інтерфейси між ними та функціональні вимоги верхнього рівня** для MVP продукту. Специфікація **не** визначає конкретний технологічний стек, мову програмування чи конкретні бібліотеки — ці рішення приймаються в окремих технічних специфікаціях (`spec-tool-*`, `spec-infrastructure-*`).

### 1.3 Цільова аудиторія

- Архітектори та розробники, що реалізують MVP.
- Product-власник (Ponomarenko) для валідації скоупу.
- Дизайнери UX/UI для опрацювання сценаріїв використання.

### 1.4 Припущення

- MVP орієнтований на український ринок (мови: UA; валюта: UAH).
- Основні пристрої споживання — мобільні (iOS/Android) та web.
- Монетизація **поза скоупом** цього документа; архітектура не повинна її блокувати.

## 2. Визначення

| Термін | Визначення |
|---|---|
| **Provider** | Користувач-постачальник послуг або товарів (майстер, магазин, бізнес). |
| **Consumer** | Користувач, що шукає послугу/товар. |
| **Listing** | Оголошення про послугу або товар (аналог OLX-оголошення). |
| **Portfolio Post** | Постійна публікація в портфоліо Provider-а (аналог Instagram-поста). |
| **Story** | Ефемерний контент із часом життя 24 год (аналог Instagram Stories). |
| **Review** | Верифікований відгук від Consumer-а за фактом завершеної угоди/послуги. |
| **Rating** | Агрегований числовий показник (1–5) на основі Review. |
| **Deal** | Зафіксована взаємодія Provider↔Consumer, до якої прив'язується Review. |
| **Feed** | Персоналізована стрічка контенту (Portfolio Posts + Stories + Listings). |
| **MVP** | Minimum Viable Product — мінімальний життєздатний продукт. |
| **UGC** | User-Generated Content — контент, створений користувачами. |
| **KYC** | Know Your Customer — процедура верифікації особи. |

## 3. Вимоги, обмеження та настанови

### 3.1 Функціональні вимоги

- **REQ-001**: Система МАЄ підтримувати два типи акаунтів: `Provider` (бізнес/майстер) та `Consumer` (замовник/покупець). Один фізичний користувач МОЖЕ мати обидва ролі одночасно під одним акаунтом.
- **REQ-002**: Provider МАЄ мати профільну сторінку, що містить: аватар, назву, опис, контакти, категорію діяльності, географію роботи, портфоліо, активні сторіс, агрегований рейтинг, кількість відгуків.
- **REQ-003**: Provider МАЄ мати можливість публікувати `Portfolio Post` з 1–10 медіа-файлами (фото/відео), підписом і тегами.
- **REQ-004**: Provider МАЄ мати можливість публікувати `Story` з автоматичним видаленням через 24 години.
- **REQ-005**: Будь-який користувач МАЄ мати можливість створити `Listing` одного з двох типів: `Offer` (пропоную послугу/товар) або `Request` (шукаю послугу/товар).
- **REQ-006**: Listing МАЄ підтримувати поля: назва, опис, категорія, підкатегорія, ціна (або діапазон, або «договірна»), локація, медіа, контактні опції.
- **REQ-007**: Consumer МАЄ мати можливість залишити `Review` лише після зафіксованої `Deal` з цим Provider-ом (верифікація факту взаємодії).
- **REQ-008**: Review МАЄ містити: оцінку 1–5, текстовий коментар, опційні медіа, опційну відповідь Provider-а.
- **REQ-009**: Система МАЄ підтримувати пошук за: категорією, геолокацією, ключовими словами, ціновим діапазоном, рейтингом.
- **REQ-010**: Система МАЄ надавати персоналізований Feed із змішаним контентом (Portfolio Posts релевантних Providers + рекомендовані Listings + Stories).
- **REQ-011**: Система МАЄ підтримувати прямі повідомлення (1-на-1) між Consumer і Provider.
- **REQ-012**: Система МАЄ підтримувати підписку Consumer-а на Provider-а (follow).
- **REQ-013**: Система МАЄ підтримувати реакції (like) та коментарі до Portfolio Posts. Коментарі НЕ вимикаються (ні на рівні поста, ні на рівні профілю).
- **REQ-014**: `Deal` ініціюється `Consumer`-ом із обов'язковими полями (опис, бюджет/ціна, дедлайн або термін виконання, опційне посилання на `Listing`). `Provider` МАЄ явно підтвердити або відхилити пропозицію. Deal переходить у статус `active` лише після підтвердження `Provider`-ом; до цього моменту перебуває у стані `pending`.
- **REQ-015**: Будь-який зареєстрований користувач МОЖЕ запропонувати нову **підкатегорію** в межах існуючої категорії (кореневі категорії створюються лише адміністратором). Пропозиція створюється у стані `pending`; після `approved` адміністратором вона додається в дерево та стає доступною всім; у стані `rejected` не використовується повторно.

### 3.2 Нефункціональні вимоги

- **NFR-001 (Продуктивність)**: Час відповіді API для 95-го перцентиля операцій читання ≤ 300 мс; для стрічки (Feed) ≤ 500 мс.
- **NFR-002 (Масштабованість)**: Архітектура МАЄ підтримувати горизонтальне масштабування сервісів стрічки, медіа та пошуку незалежно один від одного.
- **NFR-003 (Доступність)**: Цільова доступність MVP — 99,5 % на місяць.
- **NFR-004 (Мобільність)**: Фронтенд МАЄ бути responsive та орієнтованим передусім на мобільний UX.
- **NFR-005 (Локалізація)**: Інтерфейс MVP МАЄ підтримувати українську мову як єдину обов'язкову. Інші мови (RU/EN тощо) — поза скоупом MVP, розглядаються як пост-MVP план; архітектура локалізації НЕ повинна блокувати їх додавання.

### 3.3 Безпекові вимоги

- **SEC-001**: Усі з'єднання клієнт↔сервер МАЮТЬ використовувати TLS 1.2+.
- **SEC-002**: Паролі МАЮТЬ зберігатися з використанням стійкої функції гешування (Argon2 / bcrypt).
- **SEC-003**: Автентифікація МАЄ базуватися на короткоживучих access-токенах та refresh-токенах.
- **SEC-004**: Provider-акаунти, що продають товари з оплатою, МАЮТЬ пройти базовий KYC (підтвердження телефону та email, опційно документ).
- **SEC-005**: Система МАЄ передбачати механізм скарг на контент (`Report`) з модерацією.
- **SEC-006**: UGC медіа МАЄ проходити автоматичну перевірку на заборонений контент (NSFW/насильство) перед публікацією.

### 3.4 Обмеження

- **CON-001**: MVP розгортається як єдиний web-застосунок з API, що обслуговує і web-, і майбутні mobile-клієнти.
- **CON-002**: Платежі між Consumer і Provider **НЕ обробляються платформою** в MVP (зовнішня угода); платформа лише фіксує факт `Deal` для верифікації Review.
- **CON-003**: Story МАЄ автоматично зникати з публічної видимості через 24 год після публікації. Автор зберігає доступ до власного Story в **особистому архіві протягом 30 діб** з моменту публікації; після закінчення 30 діб Story видаляється безповоротно (включно з медіа в об'єктному сховищі).
- **CON-004**: Обсяг одного медіа-файлу обмежений: фото ≤ 10 МБ, відео ≤ 100 МБ, тривалість відео ≤ 60 с.
- **CON-005**: Глибина дерева категорій (`DAT-001`) обмежена **3 рівнями** (root → 2-й рівень → 3-й рівень). Пропозиції нових підкатегорій (REQ-015) відхиляються автоматично, якщо: (а) порушується ліміт глибини, (б) нормалізований `slug` або `name` дублює існуючу активну категорію або іншу `pending`-пропозицію.

### 3.5 Настанови

- **GUD-001**: Дотримуватися доменно-орієнтованого поділу на сервіси/модулі (Users, Content, Listings, Reviews, Messaging, Search, Feed).
- **GUD-002**: Стрічка (Feed) має будуватися асинхронно (fan-out on write або hybrid) — синхронна побудова при читанні лише як fallback.
- **GUD-003**: Усі дії користувача, що впливають на рейтинг чи видимість, МАЮТЬ логуватися для аудиту.
- **GUD-004**: Дизайн API МАЄ бути клієнт-орієнтованим: одна операція стрічки — один запит.

### 3.6 Архітектурні патерни

- **PAT-001**: Розділення на домени за DDD-підходом (bounded contexts).
- **PAT-002**: Event-driven комунікація між доменами для похідних даних (рейтинг, feed, сповіщення).
- **PAT-003**: CQRS для доменів з високим навантаженням на читання (Feed, Search).
- **PAT-004**: Media-pipeline з окремим сервісом завантаження, обробки (трансcoding, thumbnails) та CDN-роздачі.

```
# Користувачі та профілі

PATCH  /users/me
GET    /users/{id}/portfolio
GET    /users/{id}/stories
GET    /users/{id}/reviews

# Контент

GET    /posts/{id}

POST   /posts/{id}/likes
POST   /posts/{id}/comments

# Оголошення

GET    /listings/{id}

# Угоди та відгуки
POST   /deals                    # Consumer ініціює; стан `pending` до підтвердження Provider-ом

# Категорії
GET    /categories                         # дерево активних категорій
POST   /categories/proposals               # запропонувати підкатегорію (REQ-015)
GET    /admin/categories/proposals         # список pending (адмін)
POST   /admin/categories/proposals/{id}/approve
POST   /admin/categories/proposals/{id}/reject

# Соціальне
POST   /users/{id}/follow
DELETE /users/{id}/follow
GET    /feed                     # персоналізована стрічка

# Повідомлення
GET    /chats
GET    /chats/{id}/messages
POST   /chats/{id}/messages

# Модерація
POST   /reports
```

## 5. Критерії приймання

- **AC-001**: Given зареєстрований Provider, When він публікує Portfolio Post з 3 фото та підписом, Then пост з'являється на його профілі в топі й потрапляє у стрічку його підписників протягом ≤ 10 с.
- **AC-002**: Given Provider опублікував Story, When минає 24 год, Then Story зникає з публічної видимості, але залишається в архіві автора.
- **AC-003**: Given Consumer виконує пошук «ремонт техніки Київ», When він переходить у категорію, Then система повертає список Providers, відсортований за комбінацією релевантності, рейтингу та відстані.
- **AC-004**: Given Consumer та Provider не мають зафіксованої `Deal`, When Consumer намагається залишити Review, Then система повертає помилку `403 Forbidden` і пояснення.
- **AC-005**: Given Consumer залишив Review з оцінкою 4, When подія `review.created` обробляється, Then агрегований `ratingAvg` у `ProviderProfile` оновлюється в межах ≤ 30 с.
- **AC-006**: Given користувач скаржиться на контент, When він натискає `Report` і обирає причину, Then створюється `Report` у стані `pending` і контент автоматично знижується у видимості до модерації.
- **AC-007**: Given Provider публікує Listing без обов'язкових полів, When надсилається запит, Then API повертає `400 Bad Request` зі списком помилок валідації.
- **AC-008**: Given Consumer підписаний на 10 Providers, When він відкриває Feed, Then він бачить змішаний контент (пости, сторіс, рекомендовані listings) за ≤ 500 мс (p95).
- **AC-009**: Given Consumer створив Deal, When Provider ще не підтвердив, Then Deal перебуває у стані `pending`, не відображається як активна угода в публічних метриках і НЕ розблоковує можливість залишити Review.
- **AC-010**: Given Consumer створив Deal, When Provider натискає «Відхилити» з причиною, Then Deal переходить у стан `rejected`, Consumer отримує сповіщення та НЕ може залишити Review на цей Deal.
- **AC-011**: Given автор опублікував Story, When минає 24 год, Then Story зникає з публічної видимості; коли минає 30 діб з моменту публікації, Then Story і пов'язані медіа видаляються безповоротно (жодного доступу, включно з автором).
- **AC-012**: Given користувач надсилає пропозицію підкатегорії, яка дублює за нормалізованим `slug` існуючу `active`-категорію або іншу `pending`-пропозицію, When система обробляє запит, Then пропозиція відхиляється автоматично з кодом валідації `duplicate_category`.
- **AC-013**: Given користувач намагається запропонувати категорію 4-го рівня (parent вже на 3-му рівні), When система обробляє запит, Then пропозиція відхиляється з кодом `max_depth_exceeded`.
- **AC-014**: Given адміністратор затверджує `CategoryProposal`, When подія `category.approved` обробляється, Then нова категорія з'являється в дереві `GET /categories` та стає доступною для вибору в `Listing` протягом ≤ 60 с.
- **AC-015**: Given користувач намагається вимкнути коментарі до свого Portfolio Post, When він шукає відповідну опцію в UI/API, Then така можливість відсутня (REQ-013).

## 6. Стратегія тест-автоматизації

- **Рівні тестування**:
  - Unit: доменна логіка (правила Review, валідація Listing, розрахунок рейтингу).
  - Integration: взаємодія сервісів через подіївий шар, робота з БД, медіа-pipeline.
  - Contract: API-контракти між бекенд-сервісами та клієнтами.
  - E2E: ключові сценарії (реєстрація → публікація → угода → відгук).
- **Фреймворки**: обираються в технічній специфікації; обов'язкові вимоги — підтримка снепшотів API-контрактів і фікстур медіа.
- **Керування тестовими даними**: фабрики доменних сутностей; ізольована тестова БД на тест-ран; тестовий S3/об'єктне сховище для медіа.
- **CI/CD**: усі PR проганяють Unit + Contract; merge у `main` — Integration + E2E на staging.
- **Покриття**: мінімум 70 % по доменній логіці (Users, Listings, Reviews, Deals).
- **Продуктивність**: окремий навантажувальний прогон Feed та Search перед кожним релізом (ціль NFR-001).

## 7. Обґрунтування та контекст

Продукт націлений на закриття трьох розрізнених сценаріїв, які сьогодні вимагають від користувача використання кількох різних платформ:

1. **Інстаграм** дає чудовий візуальний соціальний доказ (портфоліо, сторіс), але не має структурованого маркетплейсу та верифікованих відгуків.
2. **OLX** дає маркетплейс і географічний пошук, але слабкий у візуальному бренд-сторі та майже не має механізму довіри через відгуки за фактом угоди.
3. **Rozetka** має зрілу систему відгуків, прив'язаних до покупок, але це закритий e-commerce, а не платформа взаємодії між людьми.

Ключова архітектурна ставка — **`Deal` як фундамент довіри**: Review може існувати тільки за наявності зафіксованої угоди. Це відрізняє продукт від Instagram (де відгуки в коментарях не верифіковані) й наближає до моделі Rozetka, але для широкого спектру послуг і товарів, а не лише для ретейлу.

Поділ на домени (GUD-001) та event-driven комунікація (PAT-002) обрані, щоб дати змогу пізніше винести Feed, Search і Media у незалежно масштабовані сервіси — саме вони першими впираються у навантаження в продуктах цього класу.

## 8. Залежності та зовнішні інтеграції

### Зовнішні системи
- **EXT-001**: Провайдер SMS/OTP — для верифікації телефону під час реєстрації та KYC.
- **EXT-002**: Email-провайдер транзакційних листів — підтвердження email, сповіщення, відновлення пароля.
- **EXT-003**: Геокодинг-сервіс — для нормалізації адрес та побудови пошуку за відстанню.

### Сторонні сервіси
- **SVC-001**: Push-нотифікації (web + mobile) — для сповіщень про повідомлення, нові відгуки, події угод.
- **SVC-002**: Автомодерація медіа — класифікація NSFW/насильства перед публікацією (SEC-006).
- **SVC-003**: CDN для роздачі медіа — низька затримка віддачі фото/відео.

### Інфраструктурні залежності
- **INF-001**: Реляційне сховище для транзакційних даних (Users, Listings, Deals, Reviews).
- **INF-002**: Об'єктне сховище (S3-сумісне) для медіа-файлів.
- **INF-003**: Шар кешування / in-memory store для Feed, сесій, лічильників.
- **INF-004**: Повнотекстовий/векторний пошуковий індекс для Listings та Providers.
- **INF-005**: Шина подій (message broker) для event-driven зв'язку між доменами.

### Залежності від даних
- **DAT-001**: Довідник категорій послуг і товарів — **гібридна модель**: (а) редакційне дерево, де кореневі та опорні категорії задаються адміністратором; (б) користувачі можуть пропонувати підкатегорії через модеровану чергу (REQ-015, CON-005). Формат: ієрархічне дерево з глибиною до 3 рівнів. Джерело істини — внутрішня БД платформи; оновлення редакційної частини — рідкісне (тижні/місяці), користувацьких пропозицій — постійно.
- **DAT-002**: Геодані (населені пункти, регіони) — із зовнішнього провайдера, оновлення щоквартально.

### Платформні залежності
- **PLT-001**: Вимога підтримки сучасних браузерів (останні 2 мажорні версії Chromium, Firefox, Safari) та майбутніх mobile-клієнтів iOS 15+/Android 10+.

### Комплаєнс-залежності
- **COM-001**: Дотримання ЗУ «Про захист персональних даних» (Україна) — обробка та зберігання ПД користувачів.
- **COM-002**: Механізм видалення акаунта та пов'язаних ПД на вимогу користувача (right-to-erasure-подібний).

**Примітка**: Конкретні постачальники (наприклад, який саме SMS-шлюз або CDN) визначаються в окремих infrastructure-специфікаціях і не є частиною цього документа.

## 9. Приклади та граничні випадки

### 9.1 Сценарій: створення Review лише після Deal

```text
1. Consumer C знаходить Provider P через пошук.
2. C і P обмінюються повідомленнями у вбудованому чаті.
3. C ініціює створення Deal із умовами (опис, бюджет, дедлайн) → Deal(status=pending).
4. P отримує сповіщення та підтверджує Deal → Deal(status=active).
   (Альтернатива: P відхиляє → Deal(status=rejected), Review недоступний.)
5. Після виконання послуги одна зі сторін ініціює закриття Deal → status=closed.
6. Тепер і тільки тепер C може залишити Review, прив'язаний до цього Deal.
```

### 9.2 Граничний випадок: Story після видалення автора

```text
Given Story S опублікована User U
When User U видаляє акаунт через 6 годин після публікації
Then S негайно видаляється з видимості та з архіву;
     жодна подія "story.viewed" після цього моменту не приймається.
```

### 9.2.1 Граничний випадок: життєвий цикл Story (24 год / 30 діб)

```text
t = 0:          Story S опублікована, публічно видима.
t = 24 год:     story.expired → S зникає з Feed та профілю;
                залишається лише в приватному архіві автора.
t = 30 діб:     story.purged → S та її медіа безповоротно видаляються
                з об'єктного сховища; автор більше не має доступу.
```

### 9.3 Граничний випадок: Review на видаленого Provider-а

```text
Given Provider P видалив акаунт
When Consumer C відкриває старий чат/deal і бачить кнопку "залишити відгук"
Then API повертає 410 Gone; UI показує "Користувач більше не доступний".
Агреговані історичні рейтинги зберігаються анонімізованими лише для аналітики.
```

### 9.3.1 Граничний випадок: пропозиція дублікатної категорії

```text
Given існує активна категорія "Ремонт побутової техніки" (slug: "remont-pobutovoi-tehniky")
When користувач надсилає POST /categories/proposals з name = "Ремонт побутової техніки "
     (з пробілом у кінці, нормалізований slug збігається)
Then API повертає 409 Conflict, code = "duplicate_category";
     CategoryProposal НЕ створюється.
```

### 9.3.2 Граничний випадок: відхилена пропозиція не переходить у pending повторно

```text
Given CategoryProposal P у стані "rejected"
When інший користувач надсилає ідентичну пропозицію (той самий parent + slug)
Then створюється нова CategoryProposal у стані "pending" (P історично зберігається,
     але не блокує нові пропозиції — лише active-категорії та поточні pending блокують дублікати).
```

### 9.4 Приклад запиту стрічки

```http
GET /feed?limit=20&cursor=eyJvZmZzZXQiOjQwfQ==
Authorization: Bearer <access_token>
```

Відповідь — змішаний масив елементів з полем `kind` ∈ { `post`, `story_group`, `listing_ad` }.

## 10. Критерії валідації

Реалізація вважається такою, що відповідає цій специфікації, якщо:

1. Усі `REQ-00X` реалізовані та покриті автоматизованими тестами.
2. Усі `AC-00X` проходять на staging до релізу.
3. `NFR-001` (час відповіді Feed p95 ≤ 500 мс) підтверджено навантажувальним тестом на синтетичному датасеті ≥ 100k користувачів та ≥ 500k постів.
4. `SEC-001…SEC-006` перевірено окремим security-review (може використовуватися вбудований `/security-review`).
5. Документовано відповідні технічні специфікації для кожного зовнішнього залежного компонента (`EXT-*`, `SVC-*`, `INF-*`).
6. Продемонстровано e2e-сценарій «реєстрація → портфоліо → пошук → чат → угода → відгук» на staging без ручних втручань у БД.
